### 二、产品背景与问题定义

#### 2.1 用户真实痛点

当前中国旅游用户在出行前普遍面临以下问题：

1. **信息分散**：需在多个平台（小红书、抖音、知乎等）反复查攻略  
2. **噪音干扰**：广告、软文、商家推广混杂，难以分辨真实体验  
3. **匹配度低**：攻略未考虑用户实际出行场景（如带娃、情侣、独行等）  
4. **决策割裂**：景点、餐饮、住宿、交通信息分散，无法一次性完成行程规划  
5. **打卡体验差**：拍照点信息零散，常遇人多、位置差、参考价值低等问题  

#### 2.2 我们不解决什么

本产品**不试图提供更多信息**，也**不替代所有平台**。明确避免：

- ? 做信息流社区  
- ? 做商家推荐平台  
- ? 做“我觉得你该去”的主观建议  

#### 2.3 核心目标

> **在不制造信息噪音的前提下，提取对旅游真正有用的信息，让用户不做攻略，也能直接出发。**

### 三、产品定位与核心理念

#### 3.1 产品定位  
一个面向中国所有旅游用户的 **「智能行程规划内容平台」**。

#### 3.2 核心理念  

- 不评价“值不值得去”  
- 不告诉用户“该怎么选”  
- **只回答一个问题**：  
  > “和你出行情况相似的人，去了之后，真实体验如何？”

我们呈现的是**客观、可验证的实际体验结果**，而非主观推荐。

---

### 四、目标用户定义

#### 4.1 用户范围  
所有中国旅游用户，**不设固定人群标签**。

#### 4.2 唯一差异来源  
用户差异仅来源于**当次出行场景**，而非身份。核心维度包括：

- 出行方式：独行 / 团队  
- 人数与关系类型：情侣 / 家人 / 朋友 / 其他  
- 是否带娃（高权重）  
- 预算区间（可选）  
- 出行节奏偏好：特种兵 / 轻松 / 出片 / 吃货 / 风景等  

---

### 五、核心系统一：出行画像卡片系统（P0）

#### 5.1 系统定义  
出行画像卡片是系统理解用户的**唯一结构化抓手**，作用等同于电商中的「收货地址」。

- 用户可创建、保存多个卡片  
- 每次出行只需选择对应卡片  

#### 5.2 卡片生成方式  

**方式一：自然语言生成（主路径）**  
提供大模型输入框，提示用户：  
> “请用一句话描述你的出行情况。”

**示例**：  
- “我想带老婆孩子去西安玩三天，不想要太累的特种兵行程，多推荐点拍照好看的地方。”  
- “我准备带女朋友去阳朔玩。”  
- “我和朋友四个人去重庆。”  

系统自动解析并生成结构化画像卡片。

**方式二：结构化编辑（辅助）**  
用户可手动微调人数、关系、预算等字段。

#### 5.3 卡片字段说明  

- 出行模式（独行 / 团队）  
- 人数  
- 关系类型  
- 是否带娃  
- 预算区间（可不填）  
- 系统生成的偏好标签  
- 创建时间 / 最近使用时间  

---

### 六、出发地获取规则

出发地用于：路线规划、出发时间计算、交通方式判断。

**获取方式**：  
- 系统定位自动获取（推荐）  
- 用户手动选择城市  

> 定位权限非强制，但会显著提升体验。

---

### 七、核心系统二：动态行程生成与「导演引擎」（P0）

#### 7.1 行程生成基础逻辑  

系统基于以下信息生成完整行程：  
- 出行画像卡片  
- 出发地  
- 目的地  
- 出行天数（可不填）  
- 实时天气 / 路况  

**生成内容包括**：  
- 景点顺序与路线  
- 交通方式（公交 / 打车 / 自驾）  
- 拥堵与停车提示  
- 天气与季节说明（仅提示，不建议延期）  
- 吃 / 住 / 外卖建议  
- 不同预算下的消费区间（若未填预算，提供默认参考值并提示用户）  

**行程完成后**：  
- 自动外链购票平台（机票 / 火车票）——按预算匹配  
- 自驾用户直接外链导航  

#### 7.2 自驾与停车逻辑  

结合当天时间与天气，提供**多个目的地选择策略**，例如：  
> “景区附近停车困难，建议停在 X 公里外换乘公交/打车。”

明确提示：  
- 停车难易度  
- 收费区间  

并提供两种方案：  
- 性价比优先  
- 舒适度优先  

#### 7.3 实时修正与「导演模式」（V2）  

系统在行程执行中具备**动态平替能力**。  

**意图识别示例**：  
- “人太多了”  
- “想走了”  
- “找个清静点的地方”  

系统将基于当前位置与画像，给出 **2–3 个可选平替点**（无合适选项则不提供，绝不硬凑）。  

**平替类型**：  
- 室内安静场所  
- 户外风景点  
- 临时休息区  

**导演模式原则**：  
- 不替用户做决定  
- 永远提供选择  

**示例回复**：  
> “这儿确实有点吵，附近有 A（室内安静）和 B（江景），想去哪？晚饭也可以一起顺路安排。”

### 八、核心系统三：交互与执行系统（P0）

#### 8.1 一键直达（地图跳转）  

**硬性要求**：  
所有吃、喝、住、行、平替点**必须挂载地图跳转链接**。  

**技术实现**：  
封装高德 / 百度 / Google URL Scheme  

**目标**：  
点击名称即可导航，彻底消除“找不到”的焦虑。

---

### 九、图片系统设计

#### 9.1 当前阶段（P0）  
- 使用外链图片，**不存储图片文件**  
- 图片来源：  
  - 官方景点图  
  - 餐厅与菜品图  
  - 拍照点示意图  

#### 9.2 后期升级（P1）  
- 优先展示“已验证帖子”的**真实用户图片**  
- **不改变数据结构**，仅替换展示来源

### 十、评价与打卡系统（P1）

**评价方式**：  
- ?? 喜欢  
- ? 强烈推荐（下次还会来）  
- ?? 不满意  （**不对外展示，仅用于计算好评率**）  
- ?? 极度不满（**不对外展示，仅用于计算好评率**）  

**默认规则**：  
- 去过但未评价 → 默认好评（需提示用户）  

**已打卡机制（防霸榜）**：  
- 用户点击「已打卡」后，该内容不再出现在主列表  
- 仅在「我的打卡」中可查看  

---

### 十一、帖子系统（P2，严格限定）

#### 11.1 内容范围  

**仅允许**：  
- 景点拍照位置  
- 拍摄方式（角度、设备、时间等）  

**明确禁止**：  
- 酒店推荐  
- 餐厅推荐  
- 商业广告  

#### 11.2 帖子类型  

- 未验证（免费）  
- 已验证（可收费）  

---

### 十二、商业模式（P2）

- **用户端**：10 元买断永久使用  
- **创作者端**：作者可设置单帖收费；平台收取少量服务费  
- **合作分成**：通过外链（酒店、车票、景点门票）与第三方谈合作分佣  

**验证条件**：  
- 验证期：30 天  
- 好评率 ≥ 80%  
- 踩比例 ≤ 20%  

**收费与结算**：  
- 验证期结束后才结算  
- 阶梯抽成：好评率越高，平台抽成越低  

---

### 十三、系统架构与 AI / 业务边界

#### 13.1 大模型（LLM）职责（仅限三项）  

1. 语义解析（自然语言 → 画像卡片）  
2. 内容去噪与干货提取  
3. 动态行程文字编排  

#### 13.2 业务系统（硬规则）职责  

- 验证期与结算逻辑  
- 帖子分类与审核  
- 已打卡去重  
- API 调度（天气 / 路况 / 地图）  

> **当 LLM 输出与事实 API 冲突时，硬规则拥有最高解释权。**

#### 13.3 动态自洽机制  

- 暴雨 / 景区关闭 → 强制拦截户外建议  
- 踩数据激增 → 自动触发风险标记  
- 下一次行程生成**必须包含避坑提醒**  

#### 13.4 4H4G 服务器优化路径  

- 云端大模型进行知识蒸馏  
- 本地部署轻量模型（7B/8B）仅做“填空式生成”  
- 实现秒级响应，低算力消耗

### 十四、开发优先级建议

| 优先级 | 功能模块 |
|--------|--------|
| **P0（核心）** | 出行画像卡片、带地图导航的行程生成、基础平替与导演逻辑 |
| **P1（感知）** | 实时天气/路况、点赞/打卡反馈闭环 |
| **P2（商业）** | 帖子验证、收费分账系统、会员体系 |

---

### 十五、产品原则总结（助理职业操守）

- 基于完整行程做调整  
- 提供选择，而非决策  
- 永远带导航入口  
- 走在用户前面，但不替他走路
